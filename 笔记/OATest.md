# OATest

### 1. 研究背景与问题

- **重要性**：深度学习编译器（如TVM、ONNXRuntime）对于将DL模型高效部署到各种硬件上至关重要，但其自身可能存在错误，尤其是**优化（Optimization/Pass）部分**——这类模型优化通常被设计为匹配**特定的计算图**结构，这意味着优化代码对**输入模式有很强的预设和依赖**，任何未预料到的输入模式都可能触发边界情况下的错误。
- **现有方法的局限**：
  - **基于语法的方法**（如NNSmith）：能生成语法上有效的计算图，但**没有针对性**地生成能够触发特定优化的测试模式，导致检测优化错误的效果有限。
  - **基于大语言模型（LLM）的方法**（如WhiteFox）：试图从源代码中挖掘优化模式，但受限于LLM对复杂代码的理解能力和“幻觉”问题，导致**模式提取不准确、生成的测试有效性低**。

### 2. OATest 技术方案

OATest 的核心思想是**“计算图合成”**，即从编译器自带的、用于测试优化正确性的**文档化测试**中提取优化感知的模式，然后将这些模式合成到多样化的计算图“上下文”中，从而生成既能触发优化又具有多样性的测试用例。其流程分为三个主要步骤：

1. **模式提取**：
   - 通过**代码插桩**动态运行编译器的文档化测试，收集每个优化所作用的计算图。
   - 根据优化的粒度（块级或子图级），从计算图中提取出对应的**优化感知模式**。这种方法比依赖LLM分析源代码更确定、更准确。
2. **计算图合成**：
   - 这是OATest解决的关键技术挑战。简单地将提取的模式插入到一个“种子图”（作为上下文）会导致模式中的算子输入未定义，从而生成无效的计算图。
   - OATest设计了两种合成策略来建立模式与上下文之间的有效连接：
     - **复用现有节点**：在上下文中寻找与模式中未定义输入兼容的已有张量变量进行连接。
     - **创建新节点**：当上下文中没有兼容变量时，通过插入填充（Padding）、重塑（Reshape）、重类型（Retype）等算子作为“桥梁”，生成兼容的变量进行连接。
3. **测试预言**：
   - 使用两种广泛接受的测试预言来判定是否发现编译器错误：**崩溃**和**结果不一致**。

### 3. 实验评估

论文在两个广泛使用的DL编译器（**TVM** 和 **ONNXRuntime**）上对OATest进行了全面评估。

- **有效性（Bug检测）**：
  - 在为期两周的模糊测试中，OATest共发现了**56个此前未知的错误**，其中**42个被开发者确认，24个已被修复**。
  - 在与最先进技术（NNSmith, WhiteFox）及自行实现的LLM基线对比的12小时测试中，OATest发现的错误数量（包括优化相关错误）**显著超过**所有基线方法。
- **代码覆盖**：
  - 在优化相关代码的**行覆盖率和分支覆盖率**上，OATest**显著高于**所有基线方法，平均分别高出66.98%和60.20%，证明其能更深入地探索编译器优化代码。
- **效率与有效性分析**：
  - OATest在给定时间内生成的**有效测试数量**远超基于LLM的方法（WhiteFox等），与基于语法的方法（NNSmith）相当甚至更多。
  - 消融实验验证了其**自适应的模式提取策略**和**两种图合成策略**对最终效果均有重要贡献。

### 4. 主要贡献

1. 提出了**计算图合成**的新思路，用于生成针对DL编译器优化的测试。
2. 实现了**OATest**技术，能够从文档化测试中提取优化模式，并通过创新的合成策略将其与多样上下文结合。
3. 通过大规模实验证明了OATest的优越性，并在两大主流DL编译器中发现了大量真实、已被确认的优化错误。

**总结**：该论文针对当前DL编译器测试技术在触发优化错误方面的不足，提出了一种基于“模式提取与合成”的、更直接且高效的方法（OATest）。该方法规避了LLM的不确定性，利用编译器自身的高质量测试作为知识源，通过创新的图合成技术生成大量有效的优化感知测试，在错误发现和代码覆盖率方面均显著超越了现有技术。

## 问题 1：新图合成是怎么做的，每个子图是怎么来的？

**1. 子图（Pattern）的来源：**

- **提取自文档化测试（Documented Tests）：** OATest 并不是随机生成子图，而是从深度学习编译器（如 TVM）现有的测试用例中提取。这些测试用例由开发人员编写，天然包含了能够触发特定优化（Pass）的**关键计算图结构**。
- **运行时捕获：** 通过代码插桩（Instrumentation）技术，在测试执行时捕获 `(计算图, 优化项)` 对，并根据优化级别（如算子融合、子图替换等）切分出对应的 **Pattern**。

**2. 合成过程：**

- **种子图（Seed Graph）作为背景：** 使用类似 NNSmith 的方法随机生成多样化的、符合语法规则的计算图，作为“背景板”。
- **模式嵌入：** 将提取出的 Pattern 插入到种子图的随机位置。
- **核心逻辑：** 这种合成方法结合了**文档测试的“针对性”（保证能触发优化）\**和\**随机生成的“多样性”（增加覆盖面）**，从而产生大量既有意义又复杂的测试用例。

## 问题 2：合成的过程如何避免未定义行为（形状对齐、类型对齐）？

为了确保合成后的图在逻辑上合法且可执行，OATest 采用了**“先抽象、后对齐”**的策略：

**1. 抽象化处理（Abstraction）：**

- 在 Pattern 插入前，抹除其具体的形状（Shape）和数据类型（Type）数值。
- **目的：** 解除原始测试用例中死板的约束（如必须是 `2x4` 矩阵），使 Pattern 变成一个“可伸缩的模板”，更容易适配新的环境。

**2. 兼容性对齐策略（双重保险）：** 当 Pattern 插入种子图产生“未定义”输入时，通过以下两种方式修复：

- **策略一：重用（Reuse）—— 寻找现成的：**
  - **做法：** 遍历合成点附近的变量池（Seed Graph 中已有节点的输出），寻找维度、类型等基本属性与 Pattern 输入要求**兼容**的变量。
  - **匹配依据：** 复用 NNSmith 等工具的规则集进行静态检查。如果找到，直接建立连接。
- **策略二：桥接（Bridge）—— 暴力改造（当重用失败时）：**
  - **做法：** 如果找不到现成的兼容变量，则随机选一个变量，通过插入**转换节点（Bridge Nodes）**强行使其合法。
  - **改造步骤：** 1.  **Padding/Cropping（填充/裁剪）：** 补齐或削减元素总量。 2.  **Reshape（重塑）：** 改变维度和轴的大小，使其与 Pattern 需求完全一致。 3.  **Cast/Retype（类型转换）：** 强制转换数据类型（如 `float32` 转 `int32`）。

**总结：** 这种方法不是盲目随机插入，而是通过**“有目标的改造”**，确保生成的每一张新图都能通过编译器的静态检查并正确运行，将合法率从原始插入的 5% 提升到了接近 100%。
