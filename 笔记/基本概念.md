# 计算图

计算图是深度学习框架和编译器中用于表示神经网络模型的核心数据结构。它是一个**有向无环图**，其中**节点**代表运算操作，**边**代表在操作之间流动的数据。

## 组成部分

### 张量（Tensor）
张量是计算图中的**数据载体**，在图中通过**边**来表示。

- **定义**：多维数组，是标量、向量、矩阵的高维推广。
- **关键属性**：
    - **形状**：定义张量的维度信息，如 `[batch_size, channels, height, width]`
    - **数据类型**：定义元素的类型，如 `float32`, `int64`, `bool` 等
    - **设备**：标识张量所在的计算设备（CPU/GPU）
- **在计算图中的角色**：
    - **输入张量**：模型的输入数据和权重参数
    - **中间张量**：各层之间的传递结果
    - **输出张量**：模型的最终预测结果

### 算子（Operator）
算子是计算图中的**计算单元**，在图中通过**节点**来表示。

- **定义**：执行特定数学运算的基本单位，也称为"操作"或"层"
- **分类**：
    - **按功能**：
        - 数学运算：`Add`, `MatMul`, `Conv2D`
        - 激活函数：`ReLU`, `Sigmoid`, `Softmax`
        - 规约操作：`Sum`, `Max`, `Mean`
        - 形状变换：`Reshape`, `Transpose`, `Concat`
    - **按输入输出关系**：
        - **逐元素运算**：输出形状与输入相同，如 `ReLU`
        - **形状改变运算**：输出形状与输入不同，如 `Reshape`
        - **多输入运算**：需要多个输入张量，如 `Concat`
- **属性**：控制算子行为的参数，如卷积的 `kernel_size`, `stride`, `padding`

### 计算图的完整结构
```
输入张量 → 算子节点 → 中间张量 → 算子节点 → ... → 输出张量
    ↑          ↑          ↑          ↑              ↑
  边(数据流)  节点(计算)  边(数据流)  节点(计算)     边(数据流)
```

## 为什么DL编译器要使用计算图

DL编译器使用计算图作为中间表示，主要基于以下几个关键原因：

### 1. **全局优化视角**
计算图提供了模型的**全局视图**，使编译器能够：
- **跨层优化**：识别可以合并或重排的操作序列
- **算子融合**：将多个小算子合并为一个大算子，减少内核启动开销和内存访问
  *示例：将 `Conv2D → BatchNorm → ReLU` 融合为单个算子*
- **常量折叠**：在编译时预计算恒定不变的子图

### 2. **自动微分与梯度计算**
计算图的DAG结构天然支持**反向模式自动微分**：
- **前向传播**：按拓扑顺序执行图中的节点
- **反向传播**：按逆拓扑顺序计算梯度，通过链式法则自动求导

### 3. **硬件无关的优化**
计算图作为**高级中间表示**，允许编译器：
- **与硬件解耦**：在同一计算图上为不同硬件后端生成优化代码
- **统一的优化通道**：在 lowering 到具体硬件之前应用通用优化

### 4. **内存优化**
通过分析计算图的**数据流关系**，编译器可以：
- **内存复用**：识别张量的生命周期，重用内存空间
- **原地操作**：确定哪些操作可以安全地原地执行，减少内存分配
- **内存布局优化**：根据硬件特性优化张量的内存排布

### 5. **并行化与调度优化**
计算图显式地表达了**数据依赖关系**，使得编译器能够：
- **依赖分析**：识别可以并行执行的操作
- **流水线调度**：合理安排计算与数据传输的重叠
- **自动并行**：在多个设备间分割计算图

### 6. **跨框架兼容性**
计算图促进了**模型互操作性**：
- **格式标准化**：ONNX等格式基于计算图，实现不同框架间的模型转换
- **统一编译接口**：不同前端框架（PyTorch、TensorFlow）可以生成相同的计算图表示

### 实际编译流程中的计算图变换
```
用户模型 → 前端计算图 → 图级优化 → 算子选择 → 后端代码生成
           (高层IR)     (计算图变换)   (硬件特定)    (可执行代码)
```

计算图因此成为连接深度学习模型定义与高效硬件执行的**关键桥梁**，使得DL编译器能够在保持语义正确性的同时，大幅提升模型推理性能。

### DL编译器架构详解

#### 核心思想：分层优化

这个架构的核心是**分层优化**。编译器将复杂的编译过程分解为多个层次，每个层次专注于一类特定的优化，使得整个系统更加模块化、可维护且强大。

整个流程可以看作是从**高级计算图** 到**底层硬件指令** 的转换与优化过程。

#### 第1层：前端 - 计算图优化

**目标**：接收来自深度学习框架（如PyTorch）的定义，构建一个计算图，并进行与硬件无关的高层优化。

这一层主要在**计算图** 层面进行操作。

##### 网络层优化
- **Operator Fusion（算子融合）**：将多个小算子合并成大算子，减少内核启动开销
- **Operator Sinking（算子下沉）**：将操作向后推，减小影响范围
- **Layout Transformation（布局转换）**：优化数据内存排列顺序
- **Memory Allocation（内存分配）**：规划张量内存生命周期，复用内存

##### 代码/计算层优化
- **Constant Fold（常量折叠）**：编译时计算常量表达式
- **CSE（公共子表达式消除）**：消除重复计算
- **DCE（死代码消除）**：删除无用代码
- **Algebraic simplification（代数简化）**：利用代数恒等式简化计算

#### 第2层：后端 - 硬件相关优化

**目标**：将优化后的计算图映射到**特定目标硬件**上，生成高效底层代码。

##### 调度优化
- **Loop Optimization（循环优化）**：循环分块、展开、交换、向量化
- **Instructions Optimization（指令优化）**：选择高效硬件指令
- **Memory Optimization（内存优化）**：优化数据访问模式

##### 自动调优技术
- **Search-based auto-tuning**：基于搜索的自动调优（如TVM Ansor）
- **Polyhedron-based auto-tuning**：基于多面体模型的自动调优

#### 第3层：内核库

**目标**：提供高度优化的基础算子例程

- 手动精心优化的核心算子
- 与编译器生成代码混合使用
- 调用专业库（如cuDNN、oneDNN）

#### 目标硬件平台

- **CPU**：x86, ARM, RISC-V - 注重指令级并行
- **GPU**：NVIDIA, AMD - 注重大规模数据并行
- **ASIC**：TPU, NNP - 专用AI芯片
- **FPGA**：Xilinx, Altera - 可配置硬件

#### 架构优势

1. **前端**："做什么"的优化，图级变换简化计算
2. **后端**："怎么做"的优化，硬件相关优化
3. **内核库**：性能基石，提供优化算子
4. **自动调优**：应对复杂硬件环境

# 算子与规约

## **一、 核心概念**

1.  **算子**
    *   **是什么**：计算图中的基本计算单元（节点），如卷积、矩阵乘法、加法。
    *   **关键特性**：
        *   **逐元素算子**：无数据依赖，易并行（如 ReLU）。
        *   **规约算子**：有数据依赖，难并行（见下文）。

2.  **规约**
    *   **是什么**：一类特殊的算子，通过聚合操作**降低数据维度**。
    *   **典型操作**：求和、求最大值、求平均值。
    *   **核心特征**：存在**数据依赖**（对同一内存位置的累加），是性能优化的重点和难点。

## **二、 核心关系与优化**

算子和规约的组合构成了深度学习模型的计算骨架，其优化是AI编译器的核心任务。

**1. 核心优化技术：算子融合**

这是最重要的优化，没有之一。

*   **目标**：将多个小算子（特别是**规约+逐元素**算子）合并成一个复合大算子。
*   **为何有效**：
    *   **减少内存读写**：避免中间结果写回慢速内存，直接在高速度缓存或寄存器中完成计算。
    *   **提升计算效率**：减少内核调用开销，提高硬件利用率。

*   **经典融合模式**：
    *   `矩阵乘法 (规约) + 偏置加法 (逐元素) + ReLU激活 (逐元素)`

**2. 针对规约的专项优化**

*   **循环变换**：对包含规约的循环进行**分块**，使数据块能放入高速缓存，极大提升数据复用。
*   **并行策略**：采用**树状规约**等算法，将大规约任务分解为小任务并行计算，再合并结果。

**3. 自动调优**

由于最优的融合方式和参数因硬件和模型而异，DL编译器（如TVM）使用**自动调优**在巨大参数空间中搜索最佳方案。

## 三、 「算子与规约」与「DL编译器」
好的，这是一份将 **「算子与规约」** 与 **「深度学习编译器」** 紧密结合的核心笔记，清晰地展示了理论概念如何转化为编译器的实际优化技术。

### **「算子与规约」在DL编译器中的核心角色**

可以将DL编译器的工作流程看作是对算子和规约的持续优化：

1.  **输入**：接收由各种算子组成的计算图。
2.  **前端优化**：识别图中的**规约**等关键算子，与相邻算子进行**图级融合**，生成更简洁、高效的融合后计算图。
3.  **后端优化**：将融合后的图转换为低层IR，并针对包含**规约**的复杂循环进行**循环变换**和**自动调优**，生成高度优化的内核代码。
4.  **输出**：在目标硬件上高效执行的机器码。

**核心思想**：DL编译器通过理解**算子**的语义和**规约**的特性，将高级计算描述**层层转化**和**协同优化**，最终解决内存瓶颈，释放硬件性能。

# Bug

**“这个阶段最核心的贡献就是定义了什么是Bug：以 differential testing 手段确定 test oracle。”**

1. 核心问题：什么是Bug？

   + 在传统软件中，Bug 是程序行为与“预期正确行为”的偏差。但这个“预期正确行为”在深度学习框架中极难定义。框架本身就是一个极其复杂的程序，没有一个简单的“正确”标准。

   + 例如，对于一个卷积操作，什么算是对的？你无法像 1+1=2 这样写出一个绝对的、简单的正确结果。

2. 解决方案：**Differential Testing（差分测试）**

   + 既然没有一个绝对的“上帝”告诉我们正确答案，那就通过比较来发现不一致。

   + 基本思想是：准备同一个输入（如一个随机生成的神经网络模型），让多个不同的深度学习框架（如 PyTorch, TensorFlow, ONNX-Runtime）去执行。如果它们的输出结果不一致，那么其中至少有一个存在Bug。

这里的 Test Oracle（测试预言） 不再是某个绝对的“正确值”，而是 “多个实现的行为一致性”。不一致就是Bug。
